# Contributing to Sandbox Probe

Thanks for taking the time to contribute!

`sandbox-probe` is Apache 2.0 licensed and accepts contributions via GitHub pull requests.

These are guidelines, not laws. We hold the bar a little higher than usual because this is a security tool — but
don't let that put you off opening a PR. We're happy to iterate with you to get a change over the line. Suggestions
for improving this document are welcome too.

## I Don't Want To Read This Whole Thing I Just Have a Question!!!

If you just have a question, the fastest path is to open an issue with the question label at
[controlplaneio/sandbox-probe/issues/new](https://github.com/controlplaneio/sandbox-probe/issues/new). Someone will
chime in.

## What Should I Know Before I Get Started?

`sandbox-probe` fingerprints the execution environment of AI code assistants (Claude Code, Gemini CLI, and similar) —
detecting sandbox boundaries, writable paths, network capabilities, processes, and so on. See the
[README](../README.md) for the user-facing overview.

There are two repositories that work as a pair:

- **[controlplaneio/sandbox-probe](https://github.com/controlplaneio/sandbox-probe)** — this repo: the probe binary,
  test harness scripts under `tests/`, and wrappers under `scripts/`.
- **[controlplaneio/sandbox-probe-reports](https://github.com/controlplaneio/sandbox-probe-reports)** — the published
  log of default sandbox permissions for popular coding agents, generated by running this probe against them.

If you're adding a new probe, a new sandbox detector, or support for another code assistant, that work lives here. If
you're publishing the results of running the probe against a new agent or release, that goes in `sandbox-probe-reports`.

## How Can I Contribute?

### Reporting Bugs

This section guides you through submitting a bug report. Following these guidelines helps maintainers and the
community understand your report, reproduce the behaviour, and find related reports.

Before creating a bug report, please check the list below — you may find you don't need to file one. When you do
file, please [include as many details as possible](#how-do-i-submit-a-good-bug-report). Fill out the issue template
for bugs.

> **Note:** If you find a **Closed** issue that looks like what you're hitting, open a new issue and link to the
> original from the body of the new one.

#### Before Submitting a Bug Report

- **Run `sandbox-probe version`** and note the version, commit, and build date — include them in the report.
- **Perform a [cursory search](https://github.com/search?q=is:issue+user:controlplaneio+sandbox-probe)** to see if
  the problem has already been reported. If it has and the issue is still open, add a comment to the existing
  issue instead of opening a new one.

#### How Do I Submit a (Good) Bug Report?

Bugs are tracked as [GitHub issues](https://guides.github.com/features/issues/). Open one at
[controlplaneio/sandbox-probe/issues/new](https://github.com/controlplaneio/sandbox-probe/issues/new) and include:

- **A clear, descriptive title** that identifies the problem.
- **Exact steps to reproduce** — which command you ran, with which flags, in which environment (host OS, container,
  inside which AI agent).
- **The relevant artifacts.** The most useful things to attach are usually:
  - `report.json` (or whatever you passed to `--output_path`)
  - the corresponding log file from `logs/sandbox-probe-YYYY-MM-DD-HH-MM-SS.log`
  - the output of `sandbox-probe version`
- **What you observed** vs. **what you expected**, and why.
- **Context**: did the problem start happening recently (e.g. after a probe upgrade, or an agent upgrade)? Can you
  reproduce it with an older version from the [releases page](https://github.com/controlplaneio/sandbox-probe/releases)?
  Is it deterministic, or intermittent?

Please censor anything sensitive in attached reports — but check the bug still reproduces with the censored version.

### Suggesting Enhancements

Enhancements include both completely new features and small improvements to existing ones — for example a new probe
category, support for a new sandbox runtime, or support for another AI code assistant.

#### Before Submitting an Enhancement Suggestion

- **Perform a [cursory search](https://github.com/search?q=is:issue+user:controlplaneio+sandbox-probe)** to see if
  the enhancement has already been suggested. If it has, comment on the existing issue.

#### How Do I Submit A (Good) Enhancement Suggestion?

Open an issue and include:

- **A clear, descriptive title.**
- **A step-by-step description** of the suggested enhancement and the workflow it would enable.
- **Concrete examples** — copy/pasteable snippets, expected output, a fragment of `report.json` showing the new
  finding, etc.
- **The current behaviour** vs. **the behaviour you'd like**, and why.
- **Why this is useful to most users** of `sandbox-probe` and isn't better as a separate tool.
- **Prior art** — other tools that do this, if any.
- **Versions** — `sandbox-probe version` output, host OS and version.

### Development

#### Prerequisites

See [Prerequisites in the README](../README.md#prerequisites) for the canonical list (Go version, `buf`, `jq`,
`docker`/`podman`, agent CLIs). At minimum you need Go 1.25+ and `make`.

```bash
make build           # builds ./bin/sandbox-probe
make install-buf     # only needed if you're changing protobuf definitions
```

#### Adding New Tasks

1. Create a new task type under `pkg/tasks/baseline/`.
2. Implement the `Task` interface:

   ```go
   type Task interface {
       GetName() string
       Run(ctx context.Context, ti Inputs) ([]*reportv1.Finding, error)
   }
   ```

3. Register the task in `GetBaselineTasks()` in `pkg/tasks/baseline.go`.
4. Add any expected types to `pkg/tasks/tasks.go`.

#### Creating Command-Based Probes

For tasks that wrap an external system command, use the generic command-based probe pattern in
`pkg/tasks/cmd-based/`. See `pkg/tasks/cmd-based/processes.go` for working examples.

1. **Define your probe struct** with the fields it will collect:

   ```go
   type myCustomProbe struct {
       result []string
   }
   ```

2. **Implement the `CmdTask[T]` interface** — `getCommand` and `parseCommandOuput`:

   > Yes, `parseCommandOuput` is the actual method name in the codebase (missing the `t` in "Output"). It's a
   > known typo we're keeping for now to avoid a breaking API change — please match it.

   ```go
   // getCommand returns the command and arguments to execute.
   func (p *myCustomProbe) getCommand() ([]string, error) {
       return []string{"mycommand", "--arg1", "--arg2"}, nil
   }

   // parseCommandOuput parses the command output into your struct.
   func (p *myCustomProbe) parseCommandOuput(out []byte) (*myCustomProbe, error) {
       lines := strings.Split(string(out), "\n")
       // ... parsing logic ...
       return &myCustomProbe{result: parsed}, nil
   }
   ```

3. **Execute the probe** with the generic runner:

   ```go
   probe := &myCustomProbe{}
   result, err := runCmdTask(probe)
   ```

4. **Write tests** using the mock executor pattern:

   ```go
   func TestMyProbe(t *testing.T) {
       mockExec := func(_ string, _ ...string) ([]byte, error) {
           return []byte("mock output"), nil
       }
       testProbe(t, "myCustomProbe", &myCustomProbe{}, mockExec, expectedResult)
   }
   ```

#### Running Tests

```bash
make e2etests        # end-to-end tests
make fmt             # format Go code
cd api && buf generate    # regenerate Protocol Buffers
```

##### Known Working Tooling Versions

installs the **latest** of each, and every report tags the exact version it ran against (`claude=…`, `opencode=…`,
`srt=…`, `kernel=…`), so the source of truth is the report, not this table. The versions below are a representative
snapshot known to work:

#### Trialling Against Agent Sandboxes

The scripts under `./tests/` are the easiest way to run the probe inside a real agent sandbox. The README's
[Testing an agent's sandbox](../README.md#testing-an-agents-sandbox) section covers running them; here we just call
out what to keep in mind when adding or changing one:

- Each script should be runnable in isolation and write its `report.json` into `./reports/`.
- If you add a script for a new agent or sandbox, follow the existing `baseline_<agent>.sh` / `sandbox_<agent>.sh`
  naming.
- Be conservative about what the agent is asked to do — see the safety note in the README. Prefer non-interactive
  invocations that only run the probe binary.

#### Testing in AI Code Assistants

When validating a change against the real agents, the wrappers under `scripts/` are convenient:

```bash
./scripts/run-claude.sh "Execute !bin/sandbox-probe"
./scripts/run-gemini-podman.sh "bin/sandbox-probe"
```

#### Agent sandboxes with no LLM (general mock)

The `run-*.sh` wrappers above drive **real, billed** agent sessions. For CI — and for deterministic
local runs — we instead exercise each agent's sandbox with **no model call, no key, no tokens**:

- `scripts/mock-agent-api.mjs` — one zero-dependency Node server that speaks **five** wire protocols,
  routing by request path: Anthropic (`/v1/messages`), Gemini (`:streamGenerateContent`), OpenAI
  Responses (`/v1/responses`, Codex), OpenAI Chat Completions (`/v1/chat/completions`, streaming and
  non-streaming — OpenCode/Goose/Pi/gptme/Cline) and Ollama (`/api/chat`, wired for a future
  native-Ollama agent; not yet exercised by a matrix row). Each returns a canned shell
  tool call running `$PROBE_CMD`, echoing whatever tool name the request advertised and shaping the
  argument from that tool's own schema (e.g. Cline's `run_commands` takes a `commands` array), and
  answers anything else trivially — so it survives CLI churn. When an agent changes its wire shape
  enough to break it, that agent's matrix rows fail loudly — the signal to update the mock.
- `scripts/run-probe-via-{claude,gemini,codex,opencode,goose,pi,gptme,cline}-stub.sh` — point the real
  CLI at the mock via its base-URL override, run it headless with approvals bypassed so the **OS
  sandbox is the only boundary** on the probe, and (for agents that ship a sandbox) toggle it on/off.
  Each stub tags the report with the agent's own `--version` (`claude=…`, `codex=…`, …) so results are
  comparable across CLI upgrades.
- `scripts/run-probe-in-sandbox.sh` — wraps the probe directly in a keyless OS sandbox (no agent, no
  model): `srt`, `firejail`, `nono`, `podman`, `docker`, `bwrap`, `nspawn`, `gvisor`. It tags the
  report with the runtime's `--version` so a behaviour change across a tool bump is visible in the diff
  (the confining agents likewise tag their sandbox engine, e.g. `bwrap=…`/`docker=…`).

Every report also carries an `environment_detection` finding with the host kernel release/version and
OS release — captured by the probe on **every** run regardless of harness — so a change caused by a
kernel or OS upgrade (seccomp/landlock/user-namespace/gVisor-systrap behaviour all track these) shows
up as a diff rather than a mystery.
- `tests/detect_{claude,codex,gemini}.sh` — `make e2etests` entry points asserting the probe reports
  the expected `sandbox_detection` (`seatbelt` on macOS; `bubblewrap`/kernel-enforcement on Linux;
  `docker` for gemini's container). Each skips gracefully when its CLI (or sandbox dep) is unavailable.

The [`scan-matrix`](../.github/workflows/scan-matrix.yaml) workflow runs the probe across a **harness**
axis — one matrix row per way of executing it. Each row sets `family` (which setup steps run), `harness`
(the job/report name), and optionally `sandbox`/`backend`/`runtime` and `expect`. Agents that ship their
own sandbox (Claude, Codex, Gemini) have both an as-is row and a confined row; OpenCode/Goose/Pi/gptme/
Cline ship none, so those run unconfined only. All rows are keyless. bwrap-based rows install
`bubblewrap` (+ `socat`) and relax the Ubuntu 24.04 unprivileged-user-namespace restriction;
`nspawn`/`gvisor` build a small rootfs.

To add a harness (another agent or sandbox runtime): add a matrix row with the right `family`/`runtime`,
a setup step and a run step gated on it, and — if it confines the probe — an `expect` array of the
`sandbox_detection` values that prove it. The assert step is data-driven off `expect`, so nothing else changes.

#### Known Limitations

##### macOS UDP Port Scanning

UDP port scanning is **disabled on macOS** (Darwin) because of reliability issues:

- **Issue:** the current UDP scanning method relies on timeout behaviour to determine port status. On macOS, all
  ports timeout regardless of their actual state, leading to false positives.
- **Workaround:** `ScanUDP()` in `pkg/tasks/baseline/network.go` returns an empty slice on macOS.
- **Future:** OS-specific UDP scanning (e.g. via `netstat`, `lsof`, or native syscalls) is planned.

```go
// pkg/tasks/baseline/network.go
func ScanUDP(host string) []int {
    // TODO: fix usage in darwin
    // it reports all the ports because they all timeout
    if runtime.GOOS == "darwin" {
        return []int{}
    }
    // ... scanning logic ...
}
```

### Pull Requests

The PR process has a few goals:

- Maintain the quality of `sandbox-probe`.
- Fix problems that matter to users.
- Engage the community in working towards the best possible probe.
- Keep contributor review sustainable for maintainers.

To get your change considered:

<!-- markdownlint-disable no-inline-html -->

1. Follow all instructions in the PR template.
2. Follow the [style guides](#style-guides).
3. After submitting, check all [status checks](https://help.github.com/articles/about-status-checks/) are passing.
   <details>
    <summary>What if the status checks are failing?</summary>
    If a status check is failing and you believe the failure is unrelated to your change, leave a comment on the
    pull request explaining why. A maintainer will re-run the check. If we conclude the failure was a false
    positive, we'll open an issue to track it in our check suite.
   </details>

<!-- markdownlint-enable no-inline-html -->

Meeting the above gets your PR reviewed; reviewers may still ask for additional tests or changes before it's
accepted.

## Style Guides

### Git Commit Messages

- [GPG sign][commit_signing] your commits if you can.
- Follow [Conventional Commits](https://www.conventionalcommits.org).
- Present tense ("add feature", not "added feature").
- Imperative mood ("move cursor to…", not "moves cursor to…").
- First line ≤ 72 characters.
- Reference issues and PRs liberally in the body.

### General Style Guide

Install an `.editorconfig`-aware plugin in your editor, or configure it to match the `.editorconfig` file at the
repo root.

### Go Style Guide

All Go code is linted with [golangci-lint](https://golangci-lint.run/). Formatting is handled by `gofmt` — run
`make fmt` before committing.

### Bash Style Guide

We follow the [Google Shell Style Guide](https://google.github.io/styleguide/shellguide.html). Bash scripts under
`tests/` and `scripts/` are linted with [shellcheck](https://www.shellcheck.net/).

### Documentation Style Guide

All Markdown is linted with [markdownlint-cli](https://github.com/igorshubovych/markdownlint-cli).
